=== 3. Формирование Pull Request в GitHub

После завершения работы в ветке `lab3`, изменения были подготовлены к интеграции в основную ветку `main` через механизм Pull Request (PR).

Этапы:

1. Все изменения сохранены локально.
2. Выполнено `git push origin lab3` для отправки ветки на GitHub.
3. В веб-интерфейсе GitHub (https://github.com/your-username/your-repo) нажата кнопка **“Compare & pull request”**.
4. Указаны:
   - **Base**: `main`
   - **Compare**: `lab3`
   - Заголовок PR: `feat: Добавлен отчёт по лабораторной работе 3`
   - Описание: краткое резюме изменений, ссылки на скриншоты, ссылки на задачи (если есть)

[.text-center]

=== 4. Ревью и одобрение Pull Request

После отправки PR преподаватель получил уведомление и провёл ревью через веб-интерфейс GitHub.

Этапы ревью:

1. Преподаватель перешёл на страницу Pull Request в браузере.
2. Проверил изменения в файлах, включая `.adoc` и изображения.
3. Убедился, что ветка `lab3` корректно сравнивается с `main` (различия отображаются в diff-представлении).
4. Назначен как reviewer (обязательный шаг для контроля качества).



[.text-center]
image::image5.png[Добавление reviewer (преподавателя) в Pull Request, width=600]

После проверки преподаватель оставил комментарии (если были) и, при отсутствии замечаний, нажал кнопку **“Approve”**.

[.text-center]
image::image6.png[Статус Pull Request: “Approved” , width=600]

=== 5. Слияние в основную ветку (Merge)

После получения одобрения:

1. Преподаватель нажал **“Merge pull request”**.
2. Выбрана опция **“Create a merge commit”** (для сохранения истории).
3. Подтверждён merge — ветка `lab3` интегрирована в `main`.

> После слияния ветка `lab3` может быть удалена (опционально) через кнопку **“Delete branch”** в интерфейсе GitHub.


== 6. Команда Commit

Данная команда используется для добавления комментария на локальной машине. То есть фиксирует изменения в репозитории. Создаётся "снимок" текущего состояния файлов, которые были изменены. Базовый синтаксис:
[suerce, bash]
----
git commit -m "Сообщение коммита"
----
Где "Сообщение коммита" — это описание изменений, которые были внесены.

Дополнительные параметры:
. -a: автоматически добавляет все изменённые файлы в коммит.
. --amend: позволяет изменить последний коммит (например, исправить сообщение).
. --no-edit: используется с --amend для сохранения текущего сообщения коммита без изменений.
. --verbose: выводит подробную информацию о процессе коммита.

Если случайно был создан коммит с ошибкой, можно использовать команду:
[suerce, bash]
----
git reset --soft HEAD~1
----
Эта команда отменит последний коммит, сохранив изменения в рабочем каталоге.

== 7. Команда Push

Данная команда используется для загрузки на сервер изменений и локальных commit. То есть для синхронизации локальных изменений с удалённым репозиторием. Базовый синтаксис:
[suerce, bash]
----
git push <remote> <branch>
----
Где <remote> — имя удалённого репозитория (обычно origin), а <branch> — имя ветки, которую нужно запушить.

Флаги:

. --force: принудительно перезаписывает удалённую ветку (используется с осторожностью).
. --set-upstream: устанавливает связь между локальной и удалённой веткой.
. --tags: пушит все теги в удалённый репозиторий.
. --all: пушит все локальные ветки в удалённый репозиторий.
. --dry-run: показывает, что будет запушено, без фактической отправки.
. --verbose: выводит подробную информацию о процессе пуша.

== 8. Команда Pull

Данная команда используется для загрузки с сервера изменений, которые внесены ранее или другими людьми. Так же можно сочетать команды commit & push(pull) и в зависимости от того что выбрано, будет сделано или "заливка" изменений и коммитов на сервер, или скачивание изменений. Коммиты при этом будут сохранены так же локально.

Команду можно использовать для синхронизации локального репозитория с удалённым. А именно, для получения актуальной версии проекта, над которым работают несколько человек. Для быстрого обновления ветки можно использовать сочетание команд pull & push. Базовый синтаксис:
[suerce, bash]
----
git pull --rebase
git pull --no-commit
git pull --verbose
----
. Пункт --rebase позволяет избежать лишних merge-коммитов в истории проекта.
. Пункт --no-commit позволяет просмотреть изменения перед созданием коммита.
. Пункт --verbose позволяет вывести дополнительную информацию о процессе выполнения команды.
